# Parchment v0.1.10 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax. Each task ends with a verifiable deliverable.

**Goal:** Ship 18 fixes — a PDF/print legibility fix, sticky editor chrome, a serious High-Contrast invisible-text a11y fix, a broad dark-mode token sweep across embed/diagram/citation/admin surfaces, the theme-apply staleness fix, print syntax highlighting, AND a true content-splitting pagination engine (real per-page sheets + per-page orientation).

**Architecture:** Next.js 16 (App Router/RSC/Turbopack), React 19, TS6 strict, Biome v2, Tiptap v3 + Yjs, Drizzle/Postgres, Shiki + highlight.js, paged.js (to be evaluated/replaced).

## Global Constraints

- Pin newest stable for any new dep. Biome CI = `biome check .` repo-wide (a stray nested `biome.json` from a worktree breaks it — `git worktree remove --force` before checking).
- No new unconditional hardcoded colors in UI surfaces. Content inside the page must use `--page-*` tokens; chrome uses chrome tokens. Print/export path (`EXPORT_STYLESHEET`) stays intentionally white-paper.
- Dark mode has THREE contexts that must stay in lock-step: `[data-color-scheme="dark"]`, `@media (prefers-color-scheme:dark) [data-color-scheme="system"]`, `[data-page-bg="dark"]`. Any token added to one dark scheme must be mirrored in the other.
- Every contrast target: text ≥ 4.5:1, large text/UI ≥ 3:1, against its ACTUAL rendered background. Verify with computed-style probes, not screenshots (the deploy screenshot tool returns stale frames).
- One release branch `release/v0.1.10`, per-item commits, single squash-or-ff to main, tag `v0.1.10`, multi-arch publish on GH runners. Do NOT auto-push the tag — gate on user confirm after CI green.
- Version bump `0.1.9 → 0.1.10` in BOTH `package.json` and `src/lib/version.ts`; prepend a `CHANGELOG` entry in `src/lib/help/content.ts`.

## Task 8 (PAGINATION): true content-splitting pagination (#11, #12, #13, 🟠 major)

**Files:** New `src/lib/editor/pagination/` (measure + split engine), modify `src/components/editor/PageCanvas.tsx`, `PrintView.tsx`, `PageSetupDialog.tsx`, `src/lib/editor/paginate.ts`.

**Goal:** Replace the visual-only paged mode (absolutely-positioned break overlays on a continuous canvas) with REAL pagination: the document is measured and split into discrete page sheets, each a separate bordered sheet with the gutter between them, supporting per-page orientation (a page can be portrait or landscape independently). And resolve #13: paged.js never works under Turbopack (`PrintView` always hits `status:'fallback'`) — either fix the bundling or REMOVE paged.js and paginate natively.

**Approach (real, not faked):**
- [ ] Build a measuring paginator: render the read-only doc into an off-screen container at the page content width, walk block nodes, accumulate rendered heights, and emit page-break indices when a page's content height would exceed the printable area (respecting page size + margins from `PageSetup`). Split list/table nodes at row boundaries where possible; never split a single atom.
- [ ] Per-page orientation: extend `PageSetup`/page model so a page index can carry an orientation override; the paginator uses that page's width/height when measuring + the sheet renders at that orientation. Add a control (Page Setup dialog or a per-page affordance) to set an individual page's orientation.
- [ ] Paged VIEW: render each computed page as its own `.parchment-page` sheet (real separate sheets, gutter between), not overlays on one canvas. Keep continuous (non-paged) mode unchanged.
- [ ] Print/#13: drop the paged.js dependency (it's dead under Turbopack) and print via the native `@page` CSS path over the real per-page sheets, OR keep paged.js only if its bundling is genuinely fixed. Prefer removal — fewer moving parts, the native path already works. Update `PrintView` to print the computed sheets.
- [ ] Reflow on edit: re-paginate (debounced) when the doc changes so pages stay correct while editing.

**Test:** unit-test the paginator's break-index logic with synthetic block-height inputs (pure function: heights[] + pageHeight → break indices). Test per-page orientation width selection.

**Verify:** live — a long doc shows multiple real page sheets with the Word-style gutter; a mixed-orientation doc shows a portrait page next to a landscape page at different widths; print/PDF produces correctly paginated, non-blank, highlighted output.

**HONESTY GATE:** if a robust measuring paginator cannot be completed + verified in this pass, ship continuous mode as default, mark paged/orientation PARTIAL explicitly in the changelog and to the user — do NOT ship a faked visual-only version as "done" (that was the I4 mistake).

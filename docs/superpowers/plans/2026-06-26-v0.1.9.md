# Parchment v0.1.9 Implementation Plan

> **For agentic workers:** execute on the `release/v0.1.9` branch, one logical commit per item/group. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Fix 14 issues found in live testing of v0.1.7/v0.1.8 — overlay clipping/stacking, scroll architecture, theming reactivity, page legibility, trash, print, the guide changelog, auto-detect, plus two larger features (real pagination + mixed page orientation).

**Architecture:** Group items by shared root cause so one fix resolves several symptoms.

**Tech Stack:** Next 16 (App Router/RSC/Turbopack), React 19, TS6 strict, Biome v2, Tiptap v3 + Yjs, Postgres + Drizzle, vitest, paged.js.

## Global Constraints

- Branch: `release/v0.1.9`. Per-item or per-group commits. CI must be green (lint/typecheck/unit/build/e2e-a11y/verify-carry-forward) before tag.
- Honesty: no item marked DONE until verified — unit-testable items by tests; visual/scroll/overlay/print/pagination items by live browser check (local dev server if standable, else the deploy post-publish). Name PARTIALs explicitly.
- Disk-mirror safety: any editor data-model change must round-trip through the markdown serializer without churning untouched docs.
- render-pm.tsx is the PUBLIC share renderer: never emit raw HTML there except export-mode-gated + escaped.
- Version bump 0.1.8 → 0.1.9 in package.json + src/lib/version.ts + a release note in src/lib/help/content.ts.

## GROUP A — Overlay layer (fixes #1, #7, #9)

**Shared root:** overlays (toolbar dropdowns, tooltips, the slash menu) render inline and never escape their clipping/stacking context, so they get clipped by `overflow` containers, trigger container scrollbars, or paint behind in-page NodeViews.

### Item #1 — ⋯ / Editing dropdowns trigger the toolbar scrollbar
**Root:** the shared `Menu` (`menus/Menu.tsx`) panel is `position:absolute` inside `.parchment-menu-wrap` inside `.parchment-toolbar` (`overflow-x:auto; overflow-y:hidden`, globals.css:1237). The open panel adds scrollable overflow → the toolbar shows its thin scrollbar; v0.1.7's `align="end"` only re-anchored it, didn't escape the box.
**Fix:** render the two toolbar `Menu` panels in a portal to `document.body` with `position:fixed` + a measured trigger rect (mirror the `ui/Dropdown`/ContextMenu `position:fixed` pattern at globals.css:672). Apply the existing `[data-color-scheme]` to the portalled panel so dark theming still applies (see Group-A shared work). MenuBar's 7 menus keep the inline behavior (they're not in an overflow box) OR adopt the same portal — decide by whether the menubar clips (verify live).
**Files:** `src/components/editor/menus/Menu.tsx` (portal + measured position), `src/app/globals.css` (`.parchment-menu-dropdown` / a `-fixed` variant).
**Verify:** open ⋯ and Editing on a normal-width editor; assert no toolbar scrollbar appears and the panel is fully on-screen (live).

### Item #7 — top-of-page tooltips clipped by viewport top
**Root:** `.px-tip` is `position:absolute; bottom:calc(100%+6px)` (globals.css:753) — always opens upward, inline (no portal). Title-bar buttons at the viewport top → tooltip clipped.
**Fix:** give `Tooltip` viewport-aware placement — flip below (`top:calc(100%+6px)`) when the wrap is within ~40px of the viewport top. Cheapest robust path: a tiny `placement` prop (`'auto'|'top'|'bottom'`) + a one-time measure on hover/focus (or pure CSS `:has`/container-query is insufficient → use a small JS measure or default title-bar tooltips to `bottom`). Title-bar (DocTitleBar) tooltips → `placement="bottom"`.
**Files:** `src/components/ui/Tooltip.tsx`, `src/app/globals.css` (`.px-tip--below`), `src/components/editor/DocTitleBar.tsx` (pass `placement="bottom"`).
**Verify:** hover Star/Comments/History/Share at the top → tooltip fully visible below the button (live).

### Item #9 — slash menu paints behind code blocks / TOC
**Root:** V1 (v0.1.4) moved the suggestion menu mount from `document.body` into the `[data-color-scheme]` wrapper (for theming). It now shares a stacking context with in-page NodeViews (`.parchment-cb-wrapper` `position:relative; overflow:hidden`, `.parchment-toc`), so `z-index:9999` no longer wins.
**Fix:** mount suggestion menus in a **themed top-level portal**: append to `document.body` (top stacking) but set `data-color-scheme` (+ `data-high-contrast`) on the menu element itself so dark theming still resolves. This is the Group-A shared mechanism (same as #1's portal). Update the suggestion `container`/`appendTo` + apply the scheme attr.
**Files:** `src/lib/editor/extensions/{slash,wiki,cite,cairn}-suggestion.ts` + `suggestion-container.ts` (container → body-level themed portal), the scheme-attr helper.
**Verify:** open `/` directly above a code block and above the TOC → menu paints on top (live).

**Group-A shared deliverable:** a small `themed-portal` util (portal to body + copy the nearest `[data-color-scheme]`/`[data-high-contrast]` onto the portal root) reused by #1 and #9; viewport-flip helper reused by #7. One overlay approach, three symptoms fixed.

## GROUP B — Scroll architecture (fixes #12, #14)

**Shared root:** a nested scroll container in the editor view. `.parchment-editor-shell` is `min-height:100vh` (globals.css:404) and `.parchment-canvas-gutter` has `overflow-x:auto` (globals.css:826), which can create a second scroll context — producing a double scrollbar (#12) and breaking `position:sticky` chrome (#14, title/menu/toolbar sticky at top 0/56/88).

### Items #12 + #14
**Fix:** collapse to a **single scroll container = the window**. Ensure no ancestor between the chrome stack and the page content establishes a competing scroll/overflow context on the Y axis, and that `overflow-x:auto` on the gutter does not also clip/scroll Y (scope it so X-overflow only engages at high zoom, e.g. `overflow-x:auto; overflow-y:visible` and not on a `100vh`-height box). Verify the chrome's nearest scrolling ancestor is the window so `position:sticky` engages.
**Files:** `src/app/globals.css` (`.parchment-editor-shell`, `.parchment-canvas-gutter`, chrome-stack), possibly `PageCanvas.tsx`.
**Verify (live, REQUIRED — this is layout):** open a multi-page doc; assert exactly one vertical scrollbar (the window) and the title bar + toolbar stay pinned while scrolling. **Pin the exact offending ancestor in the browser before editing** (query elements where `scrollHeight>clientHeight` and computed `overflow-y` is auto/scroll).

## GROUP C — Real pagination (fixes #11, #13) — LARGEST, feature-scale

**Shared root:** the editor renders ONE continuous `.parchment-page` sheet with absolutely-positioned, semi-transparent break markers (globals.css:915–927); content is never split into real pages. Both Word-style page gaps (#13) and per-page orientation (#11) need discrete page sheets.

### Item #13 — Word-style gap between pages
**Approach (incremental):**
- **Phase 1 (visual approximation, ship-able):** in Paged mode, strengthen the boundary into a real **gutter gap** — render the inter-page seam as visible `--editor-gutter` space (e.g. a tall gutter-colored band + top/bottom page shadows) so moving between pages reads as two sheets. Still a continuous canvas underneath, but visually convincing. Lower risk; verify it "feels like Word."
- **Phase 2 (true pagination):** split content into discrete `.parchment-page` sheet elements at computed break offsets, each its own bordered/shadowed sheet with real gutter between. Major (measurement + reflow + caret/selection across sheets). Gate behind the same Paged-mode setting.
**Decision point:** start with Phase 1; only attempt Phase 2 if Phase 1 doesn't satisfy the Word-feel on live review.

### Item #11 — mixed page orientation (section breaks)
**Approach:** introduce a **section-break node** carrying `{ orientation, size?, margins? }`. Requires:
- editor node + slash command + UI to insert/configure a section break;
- the page canvas renders the following sheet(s) at the section's width (landscape = wider) — depends on Phase-2 pagination from #13;
- export/print: per-section `@page` named pages (paged.js named pages) — coordinate with #10;
- markdown round-trip: serialize the section marker as an HTML-comment/fence sentinel that parses back, churn-safe.
**Decision point:** #11 depends on #13 Phase 2. If Phase 2 is deferred, #11 is deferred with it — flag clearly; do not ship a half-working orientation.

**Group-C note:** This is the bulk of v0.1.9 risk. Build behind the Paged-mode toggle, keep continuous mode the safe default, and never regress the default editing experience.

## GROUP D — Theming & page legibility (fixes #6, #8)

### Item #6 — theme/style/light-dark changes require a manual refresh
**Root:** theme is server-rendered (`data-color-scheme` + `themeCssVars` on the layout from `getWorkspaceTheme`); every control persists then calls `router.refresh()` — unreliable in this Next 16 setup (same class as P3).
**Fix:** apply the change **client-side immediately** — on change, optimistically set `data-color-scheme` (+ `data-high-contrast`, `data-font`) and the `themeCssVars` inline style on the live `[data-color-scheme]` wrapper element, THEN persist (PUT) + revalidate in the background. Don't depend on the server re-render for the visible change.
**Files:** `src/components/settings/account-theme-handler.ts` (apply to DOM before/after PUT), `AccountThemeSelect.tsx`, the workspace AppearanceSettings save, the user-menu theme path, `src/lib/editor/theme.ts` (a `applyThemeToDom(theme)` helper).
**Verify:** change theme/scheme/font/accent from each entry point → page updates instantly, no reload; persists across reload (live).

### Item #8 — user-selectable white/dark page + full legibility/consistency
**Root + scope (per your decision — a user setting):**
- Add a **page-canvas choice** to the theme (extend `pageBg`): light/white sheets (current) OR a dark sheet.
- Add a real **dark `--page-*` palette** (`--page-bg`/`--page-ink`/`--page-ink-muted`/`--page-surface-muted`/`--page-border`) selected when dark page is chosen.
- **Page-token sweep:** every element rendered ON the page must use `--page-*` tokens, not chrome tokens — fix at minimum: inline `code` (`--border` → `--page-surface-muted`/`--page-border`, globals.css:1469), `.parchment-toc` (`--background` mix → `--page-*`, globals.css:1910), tables/callouts/blockquotes — audit all `.parchment-prose`/in-page selectors using `--surface`/`--background`/`--border`/`--foreground`.
- Code blocks follow the page: dark page → dark Shiki theme (github-dark) + dark `--code-bg`; white page → current light.
- A small setting UI (brainstorm the control first — radio/select in Appearance).
**Files:** `src/styles/tokens.css` (dark `--page-*` palette under each scheme + a `[data-page-bg="dark"]` variant), `src/app/globals.css` (token sweep), `src/lib/editor/theme.ts` + settings repo (the new pageBg option), `code-block-shiki.ts`/highlighter (theme follows page), the Appearance setting component.
**Verify (live, REQUIRED):** every combination (white/dark page × light/dark chrome) — body text, headings, inline code, code blocks, TOC, tables, links, selection all legible + consistent.

## STANDALONE ITEMS

### Item #2 — auto-detect misses Python & other sparse-syntax langs
**Root:** hljs `highlightAuto` scores Python low (most snippets < CONFIDENCE_THRESHOLD=5 → plaintext; one classified as Ruby). Measured.
**Fix:** add a **heuristic pre-pass** in `detectLanguage` before hljs: strong single-language signals classify directly — Python (`^\s*def `, `^\s*import `/`from .* import`, `:` line-ends + indent, `print(`, f-strings, `self`), and a few others hljs misses; only fall back to hljs when no strong signal. Optionally lower the threshold slightly (e.g. 5→4) with care. Keep churn-safe behavior (never persist plaintext from auto-detect; only in-session-edited blocks).
**Files:** `src/lib/editor/shiki/auto-detect.ts` (heuristics + thresholds), tests.
**Test:** unit — Python samples (short def, comprehension, class+import) → `python` (not plaintext/ruby); JS/TS/others still detect; ambiguous short text → plaintext. (The exact samples from triage become fixtures.)

### Item #3 — sidebar footer + `?` help menu unpolished
**Root:** `(app)/layout.tsx:86–94` footer is a plain stack of mismatched controls (avatar text, raw `<select>` LocaleSwitcher, round `?`, Sign out); the `?` opens a bespoke `.parchment-help-dropdown` unlike the polished `.px-menu`.
**Fix (design pass — brainstorm 2–3 directions, get the look right):** unify the footer into one cohesive cluster; route help through the shared menu component (or restyle the help dropdown to match `.px-menu`); polish the language control. Aesthetic → capture before/after screenshots live.
**Files:** `(app)/layout.tsx`, `HelpMenu.tsx`, `LocaleSwitcher`, globals.css footer/help styles.
**Verify:** live screenshots; user-facing aesthetic check.

### Item #4 — Parchment Guide release-notes → full per-version changelog
**Root:** `seed-guide-content.ts:60` titles the doc `Release notes — v{current}` and dumps the flat `RELEASE_NOTES.highlights`; no per-version structure.
**Fix:** add a structured `CHANGELOG: [{ version, notes[] }]` (newest-first) in `src/lib/help/content.ts`; backfill v0.1.0→v0.1.9 entries. `releaseNotesDoc()` renders a heading + bullets PER version. The "What's new" dialog can show the latest entry (or the changelog). Only affects fresh deploys (idempotent seed) — note in the item.
**Files:** `src/lib/help/content.ts` (CHANGELOG), `src/lib/docs/seed-guide-content.ts` (render all versions), maybe `HelpMenu.tsx` What's-new.
**Test:** unit — `releaseNotesDoc()` contains every CHANGELOG version heading.

### Item #5 — Trash nav → `/trash` is a dead placeholder
**Root:** `(app)/trash/page.tsx` is a stub (heading + "Plan E11" text); the real trash UI lives in FileManager (`view==='trash'`, TrashToolbar, restore/empty) but is unreachable.
**Fix:** wire `/trash` to render `FileManager` with the trash view active (pass an initial view, or have `/trash` redirect to `/files?view=trash` to match Recents/Shared/Starred which already use `?view=`). Prefer the redirect for consistency, OR render FileManager with `initialView='trash'` if a dedicated route is desired. Confirm `/files?view=trash` already activates the trash view in FileManager (searchParams path).
**Files:** `(app)/trash/page.tsx`, possibly `(app)/layout.tsx` nav href, `FileManager.tsx` (accept initial view if needed).
**Verify (live):** click Trash → trashed docs list + Restore + Empty trash all work.

### Item #10 — print-to-PDF is blank
**Root:** paged.js fails under the bundler → `status='fallback'`, but `@media print` (globals.css:4442) hides `.parchment-print-source` (the fallback content) and only prints `.parchment-print-pages` (empty on fallback) → blank.
**Fix:** gate the print CSS on status — print the **source** when `fallback`, the **pages** when `ready`. Add `data-print-status` to `.parchment-print-overlay` and scope the `@media print` show/hide on it. Also un-hide the source in fallback (it's `position:absolute; left:-9999px; visibility:hidden` — give it a print-visible state). Optionally investigate paged.js Turbopack failure, but the fallback fix is the guaranteed path.
**Files:** `PrintView.tsx` (status attr + source visibility), `src/app/globals.css` (`@media print` status-scoped).
**Verify (live):** print/save-as-PDF a real doc → content present (use the browser print-preview).

# Parchment v0.2.0 — feature list (single release)

Decision: **everything below ships under one v0.2.0 tag** (user-directed; not decomposed into milestones). Build order is internal/dependency-driven even though the tag is one release.

Cuts (explicitly out): saved searches · Obsidian/Notion bulk import · webhooks.

External dependency: the Postgres+pgvector image is built in a **separate repo/task** (`ghcr.io/jonathanmcohen/pgvector`); Parchment consumes it, does not build it.

Internal build order: **C → A → B → G → I(core) → F(hygiene, parallel) → H → D/E/F(backup) → J**.

---

## A — Multi-user / user management
- A1 User CRUD UI — create / invite / disable / delete
- A2 Roles & permissions — owner · admin · editor · viewer
- A3 Multi-user auth — login for N users, per-user sessions (today owner-only)
- A4 Per-document sharing ACLs (real, not just public/password links)
- A5 Invite flow — email invite → accept → set password (needs B)
- A6 Account lifecycle — deactivate · reset · transfer ownership

## B — SMTP / email
- B1 SMTP config UI in settings — host/port/user/pass/from/TLS; secret encrypted at rest, never echoed
- B2 Test-email button (validate before save)
- B3 Wire email → invites, password reset, share notifications
- B4 Unhide the Notifications settings section

## C — Separate database container
- C1 Strip embedded Postgres from the image (drop postgresql-18 + s6 postgres service); `DATABASE_URL` → external
- C2 `docker-compose.yml` — app + `ghcr.io/jonathanmcohen/pgvector` (align to that image's env/port/extension interface)
- C3 Migration path for existing all-in-one users (volume migrate OR backup→restore into new DB)
- C4 Docs + README quickstart updated for compose

## D — Instance-to-instance backup/restore (no S3)
- D1 Migrate to another instance — push/pull a full workspace over HTTP with a token (builds on /api/backup export+restore)
- D2 Selective restore (pick docs/folders) + dry-run

## E — Git sync (file-level markdown backup)
- E1 Git sync of the `/data/files` `.md` mirror to a remote — auto-commit + push on change / scheduled
- E2 Config UI — remote URL, branch, auth (SSH key or token, encrypted), commit cadence, author identity
- E3 Init repo + first-push / non-fast-forward / conflict surfacing
- E4 Sync status in settings (last commit, last push, errors)

## F — Repo hygiene + page polish (rolled in from v0.1.12)
- F1 S3 backup config in Settings — top-level Backup page · config form · encrypted DB storage · env-precedence · test-connection · schedule control · live-apply (no restart) · run+status · restore-from-S3 · admin-only + secret redaction
- F2 Word-style bigger page gap (thicken `.parchment-page-divider::before` band ~14→~32px + sheet shadows; match `pagination.css` sheets; true split-gap deferred to in-editor pagination)
- F3 README redo — user-friendly + de-stale (drop paged.js → native print; update version/feature status; quickstart + screenshots)
- F4 AGPL-3.0 LICENSE file + `package.json "license"` + SPDX note (fixes the About→License broken link)
- F5 Issue templates — `.github/ISSUE_TEMPLATE/` bug + feature (YAML) + config.yml
- F6 PR template — `.github/PULL_REQUEST_TEMPLATE.md`

## G — Security
- G1 MFA (TOTP)
- G2 OAuth / OIDC SSO (Google / GitHub / generic OIDC)
- G3 Audit log (who did what to which doc/user; admin-visible)
- G4 Login rate-limit / brute-force protection
- G5 Session-management UI (revoke devices/sessions)

## H — Collaboration
- H1 Comments + suggestion mode — threaded comments on a selection, resolve, @mention (suggestion tints already half-built)
- H2 Publish-to-web — read-only public page (reuses the share renderer)
- H3 Share-link permission levels (view / comment / edit) + expiry
- H4 Real-time presence — cursors / avatars of who's editing-viewing (Yjs presence)

## I — Ops / self-hosted
- I1 Health/ready/metrics — `/healthz` `/readyz` `/metrics` (Prometheus) + compose healthchecks
- I2 Per-user storage quota + admin usage dashboard (users, DB size, file storage, jobs)
- I3 Backup verification — scheduled restore-test
- I4 First-run setup wizard — owner + DB test + optional SMTP/S3 (`/setup` route exists)
- I5 Upgrade / migration tooling (esp. all-in-one → external pgvector)
- I6 Maintenance / read-only mode (admin toggle for upgrades/migrations)
- I7 Structured logging + log levels + opt-in telemetry
- I8 Container security scan in CI (Trivy/Grype)
- I9 Per-user data export (GDPR)

## J — Content / editor / search / API
- J1 Image/attachment upload storage (disk `/data` or S3)
- J2 Tags / labels + smart folders
- J3 Templates gallery
- J4 Semantic search (pgvector — already installed, unused)
- J5 Backlinks panel + graph view (Obsidian-style; wiki links exist)
- J6 Search operators
- J7 Import — markdown + docx
- J8 REST API + scoped API tokens (Developer settings exists)
- J9 CLI tool — `parchment` admin (backup/restore/user-create/migrate)
- J10 Focus / zen mode + writing goals (word target, reading time)
- J11 Bulk file ops + trash retention policy
- J12 Custom CSS / per-doc theme override + more built-in themes

Total: ~50 items / 10 groups. Design forks to resolve before their group builds: A2 role model + sharing granularity; G2 which SSO providers + lib; H1 comment data model + storage (DB vs sidecar `.md`); E2 git auth (SSH vs token) + conflict policy; D1 transport/auth between instances; J1 upload storage default (disk vs S3).
